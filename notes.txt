RDKit has recently added the ability to read ChemDraw binary files (.CDX). Interestingly, the biggest provider of these 
files is the United States Patent and Trademark Office (USPTO), which has been supplying them since 2001 for both patent
 applications and granted patents. Accessing this public dataset (easier in the past than today) was a perfect 
opportunity to put RDKit’s new functionality to the test.

Each week, the USPTO uploads a tar file containing the latest patent approvals and another for patent applications. I 
first filter the patents based on IPC/CPC codes relevant to the pharmaceutical sector and then extract all compounds 
found in the .CDX files attached to each patent. Alongside the compounds, I also extract relevant patent metadata, 
including:

    Title
    Abstract
    CPC/IPC codes
    External files .CDX and all compounds extracted from them, including the section of the patent where each compound 
appears
    ...

Each USPTO tar file produces a corresponding JSON file with the extracted data. All of this can be processed on a laptop
 (base M4 MacBook Air) in roughly 1.5 hours, and the resulting datasets are small enough to be hosted on GitHub Pages, 
with a separate repository created for each year. The code used for extraction will eventually be pushed to the 
repository.

This project was just for fun, but it turned out to be a really interesting exercise. I may explore ways to improve 
compound extraction in the future, but for now, I am keeping the simplest approach and updating it regularly.

The main page for the project is available at https://eloyfelix.github.io/uspto-chem/

Bear in mind that this only covers the USPTO. For broader chemical coverage from patents, including EPO and WO, 
SureChEMBL, developed by the EMBL-EBI Chemical Biology Services, is an excellent resource and has been steadily 
improving its functionality

 What you CAN optimize for GPU nodes:

  You can increase batch sizes when on GPU (GPUs have more memory and are faster):

  # On GPU node, use larger batch sizes
  python process_trials.py \
    --encode-batch-size 256 \  # Instead of 128 (default)
    --batch-size 5000 \         # Instead of 1000
    --num-workers 1             # GPU doesn't need many CPU workers

  For CPU nodes (what you have now):

  # On CPU node, smaller batches with more workers
  python process_trials.py \
    --encode-batch-size 32 \    # Smaller for CPU
    --batch-size 1000 \
    --num-workers 8             # More CPU parallelism

  So just request the GPU node and run the same command - it will automatically use the GPU! The only
  optimization you'd want is to increase batch sizes to take advantage of GPU memory.

  

  ✅ MDBX works perfectly on your NFS filesystem using the mdbx.Exclusive flag!

  What is the Exclusive flag?

  - Ensures single-process exclusive access to the database
  - Bypasses file locking issues on NFS (your NFS has local_lock=none)
  - Perfect for biobtree since it's typically single-process access during build/generate phases
  - Safe for your use case where one process builds the DB, then it's read-only accessed
